## Overview

The Ramp API enforces rate limits to ensure stability and fair usage. Exceeding rate limits results in a `429 Too Many Requests` response.

## Rate Limit Headers

Every API response includes rate limit information in the headers:

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Maximum requests allowed in the window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |
| `Retry-After` | Seconds to wait before retrying (on 429 only) |

## Best Practices

### Use exponential backoff

When you receive a `429`, wait before retrying. Use exponential backoff to avoid thundering herd issues:

```python
import time
import random

def backoff_retry(fn, max_retries=5):
    for attempt in range(max_retries):
        try:
            return fn()
        except RateLimitError:
            if attempt == max_retries - 1:
                raise
            wait = (2 ** attempt) + random.uniform(0, 1)
            time.sleep(wait)
```

### Respect `Retry-After`

```python
if response.status_code == 429:
    retry_after = int(response.headers.get("Retry-After", 5))
    time.sleep(retry_after)
```

### Batch where possible

Instead of making individual requests for each record, use batch endpoints where available:

- **GL Accounts:** Upload up to 500 at once via `POST /developer/v1/accounting/accounts`
- **Vendors:** Upload up to 500 at once via `POST /developer/v1/accounting/vendors`
- **Field Options:** Upload up to 500 at once via `POST /developer/v1/accounting/field-options`

### Use maximum page sizes

When paginating, request `page_size=100` to minimize total API calls.

### Cache frequently-accessed data

Data that changes infrequently (departments, locations, GL accounts) can be cached locally and refreshed on a schedule rather than fetched on every request.
